# Техническое задание: Управление земельными участками через Google Sheets

**Проект:** Балтланд (baltland.ru)
**Дата:** 2026-03-20
**Версия:** 1.0

---

## 1. Цель

Создать систему, в которой **Google Таблица** является основным инструментом для управления каталогом земельных участков. Все изменения в таблице автоматически синхронизируются с сайтом baltland.ru (база данных Supabase/PostgreSQL).

---

## 2. Архитектура системы

```
┌─────────────────────┐          ┌──────────────────┐          ┌────────────────────┐
│   Google Sheets     │  ──────► │   API-эндпоинт   │  ──────► │   Supabase (БД)    │
│   (источник истины)  │  webhook │   /api/sheets/   │  upsert  │   land_plots       │
│                     │  или poll│   sync            │  delete  │   land_plot_images  │
└─────────────────────┘          └──────────────────┘          └────────────────────┘
        ▲                                │                              │
        │                                │  логи                       │
        │                                ▼                              ▼
        │                        ┌──────────────────┐          ┌────────────────────┐
        │  статус sync           │   sync_log        │          │   Сайт baltland.ru │
        └────────────────────────│   (результаты)    │          │   (отображение)    │
                                 └──────────────────┘          └────────────────────┘
```

### Принцип работы

1. Менеджер редактирует данные в Google Sheets.
2. Скрипт (Apps Script) или триггер (по времени / по событию `onEdit`) отправляет изменённые строки на API сайта.
3. API сайта валидирует данные и обновляет БД (Supabase).
4. Сайт моментально отображает актуальные данные.
5. В таблицу возвращается статус синхронизации (✅ / ❌ + текст ошибки).

## 3. Источник истины

**Google Sheets → БД сайта** (однонаправленная синхронизация).

| Аспект | Решение |
|---|---|
| **Источник истины** | Google Sheets |
| **Направление потока** | Sheets → Supabase |
| **Обратная синхронизация** | Только статус sync и ID из БД |
| **Конфликты** | Нет — таблица всегда приоритетнее |
| **Ручное редактирование в админке сайта** | Запрещено для полей, управляемых из таблицы |

> **Почему Sheets — источник истины?** Менеджеры работают в таблице ежедневно. Это привычный инструмент, не требующий обучения. Все правки — в одном месте.

### 4.2 Статусы участка (столбец P)

| Значение в таблице | Действие на сайте | Поля в БД |
|---|---|---|
| `Доступен` | Участок виден на сайте | `is_active = true`, `is_reserved = false` |
| `Забронирован` | Виден, но с пометкой «Забронирован» | `is_active = true`, `is_reserved = true` |
| `Продан` | Скрыт с сайта (архив) | `is_active = false`, `status = 'archived'` |
| `Черновик` | Не публикуется на сайте | `is_active = false`, `status = 'active'` |
| _(строка удалена)_ | Участок удаляется или архивируется | `is_active = false` |

| Ячейка | Описание | Пример |
|---|---|---|

| `A3` | Режим: `auto` / `manual` | `auto` |

### 4.4 Служебный лист: «Лог»

| Столбец | Описание |
|---|---|

## 5. Уникальный идентификатор участка

### Стратегия двойного ключа

| Ключ | Назначение | Откуда |
|---|---|---|
| `cadastral_number` | **Бизнес-ключ** для сопоставления | Менеджер вводит в таблицу |
| `id` (uuid) | Технический PK в БД | Генерируется БД при первом INSERT |

**Логика:**
1. При синхронизации строки → ищем в БД запись с таким `cadastral_number`.
2. Если **найдена** → UPDATE существующей записи.
3. Если **не найдена** → INSERT новой, БД генерирует `uuid`, он записывается обратно в столбец A.
4. Столбец `ID` в таблице — только для чтения (менеджер его не редактирует).

## 6. Механизм синхронизации

### 6.1 Выбранный подход: Google Apps Script → API сайта

```
Google Sheets
     │
     │ Apps Script (onEdit trigger + time trigger каждые 5 мин)
     ▼
POST /api/sheets/sync
     │ Authorization: Bearer sk_sheets_***
     │ Body: { rows: [...], deletedCadastrals: [...] }
     ▼
API обработчик (Next.js Route Handler)
     │ валидация → upsert в Supabase → возврат результата
     ▼
Apps Script записывает статус в столбец AC/AD
```

### 6.2 Триггеры синхронизации

| Триггер | Когда | Что делает |
|---|---|---|
| **`onEdit`** | При каждом изменении ячейки | Помечает строку как «dirty» (нужна синхронизация) |
| **Time-based (5 мин)** | Каждые 5 минут | Собирает все «dirty» строки и отправляет пакетом |
| **Ручной** | Кнопка «Синхронизировать» в меню | Принудительная полная синхронизация |

### 6.3 Формат запроса к API

```json

Authorization: Bearer sk_sheets_xxxxx

{

      "description": "Ровный участок...",
      "price": 1850000,
      "area_sotok": 8.5,
      "district": "Зеленоградский",
      "location": "пос. Сокольники",
      "distance_to_sea": 3.5,
      "land_status": "ИЖС",
      "has_gas": true,
      "has_electricity": true,
      "has_water": false,
      "has_installment": true,
      "status": "Доступен",
      "is_featured": false,
      "ownership_type": "ownership",
      "image_url": "https://...",
      "bundle_id": null,
      "bundle_title": null,
      "is_bundle_primary": false
    }
  ],
  "deleted_cadastrals": ["39:03:999999:01"]
}
```


## 7. Обработка операций (CRUD + статус)

### 7.1 Добавление нового участка

```
Менеджер добавляет строку → заполняет обязательные поля →
Apps Script отправляет на API → API: INSERT в land_plots →
UUID возвращается в столбец A → Sync = ✅
```

**Валидация перед INSERT:**
- Кадастровый номер заполнен и уникален
- Цена > 0, площадь > 0
- Район и населённый пункт не пустые
- Категория земли из допустимого списка

### 7.2 Обновление существующего участка

```
Менеджер меняет ячейку → строка помечается dirty →
По таймеру отправляется на API → API: поиск по cadastral_number →
UPDATE найденной записи → Sync = ✅
```

**Сохраняются неуправляемые поля:**
- `coordinates_json`, `center_lat`, `center_lon` — не перезаписываются
- `has_coordinates`, `sync_error` — не перезаписываются
- `created_at` — не меняется

### 7.3 Удаление участка

```
Менеджер удаляет строку → Apps Script фиксирует удалённый cadastral_number →
API: UPDATE land_plots SET is_active = false WHERE cadastral_number = X
```

> **Мягкое удаление:** участок не удаляется физически, а архивируется (`is_active = false`). Это предотвращает потерю данных и сохраняет историю.

### 7.4 Смена статуса

```
Менеджер меняет столбец «Статус» → строка dirty →
API маппит статус:
  «Доступен»      → is_active=true,  is_reserved=false
  «Забронирован»   → is_active=true,  is_reserved=true
  «Продан»         → is_active=false, status='archived'
  «Черновик»       → is_active=false, status='active'
```

## 8. Валидация данных

### 8.1 На стороне Google Sheets (Apps Script, перед отправкой)

| Поле | Правило | Ошибка |
|---|---|---|
| Кадастровый номер | Формат `XX:XX:XXXXXX:XX` (regex) | `Неверный формат кадастрового номера` |
| Цена | Число > 0 | `Цена должна быть положительным числом` |
| Площадь | Число > 0 | `Площадь должна быть больше 0` |
| Район | Не пусто | `Укажите район` |
| Населённый пункт | Не пусто | `Укажите населённый пункт` |
| Категория земли | Одно из: ИЖС, СНТ, ДНП, ЛПХ | `Недопустимая категория земли` |
| Статус | Одно из: Доступен, Забронирован, Продан, Черновик | `Недопустимый статус` |
| URL фото | Начинается с `https://` или пусто | `URL должен начинаться с https://` |
| Расстояние до моря | Число ≥ 0 или пусто | `Расстояние должно быть числом ≥ 0` |

### 8.2 На стороне API сервера (дублирующая проверка)

- Та же валидация, что в п.8.1
- Проверка уникальности `cadastral_number` при INSERT
- Проверка существования записи при UPDATE
- Защита от SQL-инъекций (Supabase ORM)
- Rate limiting: не более 100 строк за один запрос

### 8.3 Подсветка ошибок в таблице

Apps Script подсвечивает ячейки с ошибками **красным фоном** и записывает текст ошибки в столбец «Ошибка sync» (AD).

## 9. Сравнение вариантов интеграции

| Вариант | Стоимость | Сложность | Надёжность | Реал-тайм | Рекомендация |
|---|---|---|---|---|---|
| **Google Apps Script + API** | Бесплатно | Средняя | ✅ Высокая | ~5 мин | **✅ Рекомендован** |
| Make (Integromat) | $9-16/мес | Низкая | ✅ Высокая | ~5-15 мин | Хороший вариант |
| Zapier | $20-49/мес | Низкая | ✅ Высокая | ~5-15 мин | Дорого |
| n8n (self-hosted) | Бесплатно | Высокая | ⚠️ Средняя | ~1-5 мин | Требует DevOps |
| Прямое подключение к Supabase | Бесплатно | Высокая | ⚠️ Средняя | Мгновенно | Сложная отладка |

### Рекомендация: **Google Apps Script + Next.js API Route**

**Почему:**
- **Бесплатно** — нет подписок
- **Привязан к таблице** — скрипт живёт внутри Google Sheets
- **Надёжно** — Google гарантирует выполнение триггеров
- **Просто отлаживать** — логи прямо в Sheets
- **Масштабируемо** — легко добавить поля в будущем
- **Не требует дополнительных серверов**

## 10. Риски и меры защиты

| Риск | Вероятность | Влияние | Мера защиты |
|---|---|---|---|
| Менеджер удалит данные случайно | Высокая | Высокое | Мягкое удаление (архивация), резервные копии таблицы |
| Дубликат кадастрового номера | Средняя | Среднее | Валидация уникальности в Apps Script + UNIQUE constraint в БД |
| API сайта недоступен | Низкая | Среднее | Retry с экспоненциальным backoff, лог ошибок, уведомление в Telegram |
| Rate limit Google API | Низкая | Низкое | Пакетная отправка (batch), не чаще 1 раза в 5 мин |
| Некорректный формат данных | Средняя | Среднее | Двухуровневая валидация (Sheets + API) |
| Потеря секрета API | Низкая | Высокое | Хранение в PropertiesService Apps Script, не в ячейке |
| Конфликт при одновременном редактировании | Средняя | Низкое | Last-write-wins (таблица всегда главная) |
| Слишком много участков (>10 000) | Низкая | Среднее | Пагинация sync, индексы в БД |

## 11. Пример бизнес-логики

### Сценарий 1: Новый участок

```
1. Менеджер Алексей открывает Google Sheets
2. Добавляет строку: кадастровый 39:03:010203:99, «Участок 10 сот. Зайцево», 2.5 млн ₽
3. Ставит статус «Черновик»
4. Через 5 минут Apps Script отправляет строку на API
5. API создаёт участок в БД с is_active=false
6. В столбце Sync появляется ✅ и UUID
7. Алексей добавляет фото, описание, меняет статус на «Доступен»
8. При следующей синхронизации участок появляется на сайте
```

### Сценарий 2: Бронирование

```
1. Клиент хочет забронировать участок
2. Менеджер меняет статус с «Доступен» на «Забронирован»
3. Sync → API ставит is_reserved=true
4. На сайте участок показывается с меткой «Забронирован»
5. Если клиент отказался — статус обратно на «Доступен»
```

### Сценарий 3: Продажа

```
1. Участок продан — менеджер ставит статус «Продан»
2. Sync → API ставит is_active=false, status='archived'
3. Участок пропадает из каталога на сайте
4. Данные сохраняются в БД для истории и отчётов
```

## 15. Требования к безопасности

1. **API-ключ** — каждый запрос должен содержать `Authorization: Bearer sk_sheets_...`
2. **IP whitelist** (опционально) — ограничить доступ к API по IP Google Apps Script
3. **Rate limiting** — не более 10 запросов в минуту, не более 200 строк за запрос
4. **Логирование** — все запросы пишутся в `sync_log`
5. **Уведомления** — при ошибках sync отправлять алерт в Telegram-канал администратора
6. **Резервное копирование** — ежедневный snapshot таблицы (Google Sheets version history)

#### 7.3 Базовая структура скрипта

В редакторе Apps Script замените содержимое `Code.gs` на:

```javascript
// =============================================
// БалтикЗемля — Google Sheets → API Sync
// =============================================

const SHEET_NAME = 'Участки';
const LOG_SHEET_NAME = 'Лог';
const HEADER_ROW = 1;

/**
 * Получить настройки API из свойств скрипта
 */
function getApiConfig() {
  const props = PropertiesService.getScriptProperties();
  return {
    url: props.getProperty('API_URL') || '',
    secret: props.getProperty('API_SECRET') || '',
  };
}

/**
 * Добавить пункт меню в таблицу
 */
function onOpen() {
  SpreadsheetApp.getUi()
    .createMenu('🔄 Синхронизация')
    .addItem('Синхронизировать все', 'fullSync')
    .addItem('Синхронизировать изменённые', 'syncDirtyRows')
    .addToUi();
}

/**
 * При редактировании — пометить строку как dirty
 */
function onEdit(e) {
  const sheet = e.source.getActiveSheet();
  if (sheet.getName() !== SHEET_NAME) return;
  
  const row = e.range.getRow();
  if (row <= HEADER_ROW) return;
  
  // Записать "⏳" в столбец AC (Sync)
  const syncCol = 29; // AC = 29
  sheet.getRange(row, syncCol).setValue('⏳ Ожидает sync');
}

/**
 * Собрать данные одной строки в объект
 */
function rowToObject(sheet, row) {
  const values = sheet.getRange(row, 1, 1, 30).getValues()[0];
  return {
    row_number: row,
    id: values[0] || null,                          // A: ID
    int_id: values[1] || null,                      // B: INT_ID
    cadastral_number: String(values[2] || '').trim(), // C
    title: String(values[3] || '').trim(),           // D
    description: String(values[4] || '').trim(),     // E
    price: Number(values[5]) || 0,                   // F
    area_sotok: Number(values[6]) || 0,              // G
    district: String(values[7] || '').trim(),        // H
    location: String(values[8] || '').trim(),        // I
    distance_to_sea: values[9] ? Number(values[9]) : null, // J
    land_status: String(values[10] || '').trim(),    // K
    has_gas: !!values[11],                           // L
    has_electricity: !!values[12],                   // M
    has_water: !!values[13],                         // N
    has_installment: !!values[14],                   // O
    status: String(values[15] || 'Черновик').trim(), // P
    is_featured: !!values[16],                       // Q
    ownership_type: String(values[17] || '').trim(), // R
    lease_from: values[18] || null,                  // S
    lease_to: values[19] || null,                    // T
    vri_id: String(values[20] || '').trim(),         // U
    bundle_id: String(values[21] || '').trim() || null, // V
    bundle_title: String(values[22] || '').trim(),   // W
    is_bundle_primary: !!values[23],                 // X
    image_url: String(values[24] || '').trim(),      // Y
    youtube_video_url: String(values[25] || '').trim(), // Z
    rutube_video_url: String(values[26] || '').trim(), // AA
    additional_cadastral_numbers: String(values[27] || '').trim(), // AB
  };
}

/**
 * Отправить строки на API
 */
function sendToApi(rows) {
  const config = getApiConfig();
  if (!config.url || !config.secret) {
    throw new Error('API_URL или API_SECRET не настроены в свойствах скрипта');
  }

  const payload = { rows: rows, deleted_cadastrals: [] };

  const options = {
    method: 'post',
    contentType: 'application/json',
    headers: { 'Authorization': 'Bearer ' + config.secret },
    payload: JSON.stringify(payload),
    muteHttpExceptions: true,
  };

  const response = UrlFetchApp.fetch(config.url, options);
  const code = response.getResponseCode();
  const body = JSON.parse(response.getContentText());

  if (code !== 200 || !body.success) {
    throw new Error('API ошибка ' + code + ': ' + JSON.stringify(body));
  }

  return body;
}

/**
 * Синхронизировать все строки с пометкой ⏳
 */
function syncDirtyRows() {
  const ss = SpreadsheetApp.getActiveSpreadsheet();
  const sheet = ss.getSheetByName(SHEET_NAME);
  const lastRow = sheet.getLastRow();
  
  if (lastRow <= HEADER_ROW) return;

  const syncCol = 29; // AC
  const syncValues = sheet.getRange(HEADER_ROW + 1, syncCol, lastRow - HEADER_ROW, 1).getValues();

  const dirtyRows = [];
  for (let i = 0; i < syncValues.length; i++) {
    const val = String(syncValues[i][0]);
    if (val.includes('⏳') || val === '') {
      const row = i + HEADER_ROW + 1;
      const obj = rowToObject(sheet, row);
      if (obj.cadastral_number) {
        dirtyRows.push(obj);
      }
    }
  }

  if (dirtyRows.length === 0) return;

  try {
    const result = sendToApi(dirtyRows);
    
    // Записать результаты
    for (const r of result.results) {
      const syncCell = sheet.getRange(r.row_number, syncCol);
      const errorCell = sheet.getRange(r.row_number, syncCol + 1); // AD

      if (r.error) {
        syncCell.setValue('❌ ' + new Date().toLocaleString('ru-RU'));
        errorCell.setValue(r.error);
      } else {
        syncCell.setValue('✅ ' + new Date().toLocaleString('ru-RU'));
        errorCell.setValue('');
        // Записать ID из БД в столбец A
        if (r.id) {
          sheet.getRange(r.row_number, 1).setValue(r.id);
        }
      }
    }

    writeLog('sync', dirtyRows.length + ' строк', 'ok', JSON.stringify(result.summary));
  } catch (e) {
    writeLog('sync', dirtyRows.length + ' строк', 'error', e.message);
    SpreadsheetApp.getUi().alert('Ошибка синхронизации: ' + e.message);
  }
}

/**
 * Полная синхронизация всех строк
 */
function fullSync() {
  const ss = SpreadsheetApp.getActiveSpreadsheet();
  const sheet = ss.getSheetByName(SHEET_NAME);
  const lastRow = sheet.getLastRow();

  if (lastRow <= HEADER_ROW) return;

  // Пометить все строки как dirty
  const syncCol = 29;
  for (let row = HEADER_ROW + 1; row <= lastRow; row++) {
    sheet.getRange(row, syncCol).setValue('⏳ Ожидает sync');
  }

  syncDirtyRows();
}

/**
 * Записать строку в лог
 */
function writeLog(action, cadastral, result, details) {
  const ss = SpreadsheetApp.getActiveSpreadsheet();
  let logSheet = ss.getSheetByName(LOG_SHEET_NAME);
  if (!logSheet) {
    logSheet = ss.insertSheet(LOG_SHEET_NAME);
    logSheet.getRange(1, 1, 1, 5).setValues([['Дата', 'Действие', 'Кадастровый номер', 'Результат', 'Детали']]);
  }

  logSheet.appendRow([
    new Date().toLocaleString('ru-RU'),
    action,
    cadastral,
    result,
    details,
  ]);
}
```

### Шаг 10. Проверка работоспособности

1. Добавьте тестовую строку в таблицу:
   - Кадастровый номер: `39:00:000000:99`
   - Название: `ТЕСТОВЫЙ УЧАСТОК`
   - Цена: `1000000`
   - Площадь: `10`
   - Район: `Зеленоградский`
   - Населённый пункт: `Тест`
   - Категория: `ИЖС`
   - Статус: `Черновик`

2. Подождите 5 минут (или нажмите в меню **🔄 Синхронизация → Синхронизировать все**)

3. Проверьте:
   - Столбец **AC** (Sync) = `✅ ...`
   - Столбец **A** (ID) = заполнен UUID
   - На сайте участок **не** виден (статус «Черновик»)

4. Смените статус на **«Доступен»** → подождите sync → проверьте на сайте

5. Удалите тестовую строку после проверки
